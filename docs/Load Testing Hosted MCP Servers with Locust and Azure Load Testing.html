Abstract

The Model Context Protocol (MCP) has emerged as the common interface by which AI agents invoke tools, read resources, and reuse prompts exposed by external services. As MCP servers move from local subprocesses to hosted, network-reachable endpoints, their operational behaviour under concurrent load becomes a first-class engineering concern — yet no off-the-shelf load-testing tooling speaks the protocol natively.

This guide presents a small, portable Python harness, built on Locust, that exercises the full MCP life cycle over the Streamable HTTP transport: initialize, notifications/initialized, tools/list, repeated tools/call, and a polite DELETE on shutdown. The harness handles both response modes (single-shot application/json and framed text/event-stream), supports stateful and stateless servers from a single base class, and accommodates three common authentication patterns (no auth, static bearer, dynamic per-request token).

The design is validated by a fifteen-minute, multi-server run in Azure Load Testing against four production endpoints — Microsoft Learn, GitHub, Context7, and the Azure DevOps remote MCP — driven from a single Locust process. The run produced 2,293 requests with three failures (0.13%) and surfaced distinct latency signatures per server that map cleanly to their backing implementations. The same project files run unchanged locally and in the cloud; secrets are resolved through Key Vault via a managed identity at runtime.

Keywords Model Context Protocol, MCP, Streamable HTTP, load testing, Locust, Azure Load Testing, Server-Sent Events, JSON-RPC, agent tooling, performance engineering.

1.Introduction

MCP servers are everywhere. Every team is exposing the tools, data, and actions their product can perform so an AI agent can reach in and use them. An agent is just an LLM running in a loop — it looks, thinks, calls a tool, looks again, until the job is done. The tools live on MCP.

MCP gives every agent and every tool the same way to talk. Instead of writing a fresh, fragile integration for each new service, a team builds one MCP server and any agent can use it. Each server offers three simple kinds of things: tools the agent can call to do something, resources it can read for context, and prompts it can reuse. That small, shared vocabulary is what lets one agent reach across many servers — docs, code, your product — without custom glue for each one.

As these servers move from stdio subprocesses to hosted HTTP endpoints serving many tenants, their concurrency profile, latency tail, and failure modes under load become operational questions teams must answer before production. This guide contributes a minimal, transport-faithful, portable load-testing harness for hosted MCP servers and demonstrates its use against four production endpoints.

1.1 Scope and Contributions

This guide restricts its attention to the tools primitive and the Streamable HTTP transport, since these are the only combination that produces interesting load-testing behaviour. The contributions are: (i) a faithful, life-cycle-complete Locust user class for MCP; (ii) a clean separation between stateful and stateless server shapes; (iii) a small auth-pattern taxonomy; and (iv) an empirical baseline across four real, production-grade endpoints.

2.Background — The MCP Life Cycle

2.1 Streamable HTTP is the load-testable transport

MCP defines two transports: stdio, where the agent spawns the server as a local subprocess and pipes JSON-RPC over its stdin and stdout, and Streamable HTTP, where the server lives on the network. This guide focuses on Streamable HTTP because it exposes a network endpoint that Locust can exercise. The stdio transport can still be stress-tested locally, but it does not map naturally to network load testing or provide an independent HTTP surface for this harness to test.

2.2 The MCP life cycle

Every agent-to-server conversation follows the same formal flow: how the connection opens, how messages are exchanged while it's alive, and how it shuts down cleanly. That flow is the MCP life cycle, and a faithful load test has to mimic it end to end — otherwise it is not testing what a real agent actually puts on the wire.

2.3 Phase 1 — The handshake

Throughout the rest of this guide the example server is Microsoft Learn, hosted at https://learn.microsoft.com/api/mcp. Every arrow in the following diagrams is a POST to that URL.

Before the agent can call a single tool, it has to introduce itself and let the server introduce itself back. Three messages, in order, every time:

For stateful servers, the session id is the value the agent must preserve. Every subsequent tools/call rides on the same Mcp-Session-Id header until the connection ends — which is exactly what a Locust virtual user has to do too.

One caveat: not every MCP server mints a session. Stateless servers such as Context7 and the Azure DevOps Remote MCP skip the Mcp-Session-Id header entirely — the handshake still happens, but later tools/call requests are standalone POSTs and the harness skips DELETE cleanup. GitHub is bearer-authenticated and session-bearing in this harness; Microsoft Learn is shown with a session id because that was the behaviour observed in the demo run reported here. Verify current service behaviour before treating it as a permanent contract.

2.4 Phase 2 — Discover available tools

Once the handshake is done, the agent doesn't yet know what tools the server actually exposes. It asks. The answer is the menu the LLM gets to pick from on every turn.

The response is a flat list of tool descriptors — name, human-readable description, and a JSON Schema for the arguments. A real agent feeds this list straight into the LLM's system prompt so the model knows what it can call. A load test does not need the LLM in the loop, but it still needs to call tools/list at least once per session: it is how the harness learns the real tool names instead of hard-coding them.

2.5 Phase 3 — Call a tool

The tools/call request is the primary measurement target. The agent picks a tool from the menu, fills in the arguments, and POSTs a tools/call. The response comes back in one of two shapes — the client asks for both with Accept: application/json, text/event-stream and the server picks one per request.

The request is ordinary JSON-RPC:

The client asks for both response shapes with a single Accept: application/json, text/event-stream header, and the server picks whichever it prefers per request. The harness therefore must check Content-Type first and choose how to read the body — calling .json() on an SSE response will throw, and assuming a single mode means the harness will work against some servers and fail against others.

Mode A — application/json. The body is the JSON-RPC payload. One response, no wrapping.

Mode B — text/event-stream. The body is a stream of frames. Each frame is two lines — event: message and data: <json> — and the server can send several before closing.

Same JSON-RPC result either way. The difference is the wrapper: application/json hands you one body to parse; text/event-stream hands you a stream you have to read, split into frames, and pull the data: values from. Calling .json() on an SSE response will throw. Pick one mode and the harness works against some servers and breaks against others. So always check Content-Type first, then choose how to read the body.

2.6 Phase 4 — Terminate stateful sessions

When a stateful client is done, it tells the server to terminate the session. One DELETE with the session id lets the server release the associated state.

Stateless servers do not need this step because there is no session to delete, so the agent simply stops sending requests. For stateful servers, skipping the DELETE is usually recoverable because sessions eventually time out, but explicit termination releases server-side state sooner. A polite client always cleans up.

2.7 Resources and prompts use the same message model

MCP defines two other primitives alongside tools: resources (read-oriented data the agent can pull into context, sometimes generated dynamically) and prompts (reusable templates the user can invoke). Both follow the same JSON-RPC + JSON/SSE shape shown above.

This guide focuses on tools only. Tool calls are the primary workload for this harness because they exercise the highest-cost and highest-risk actions agents invoke repeatedly. Resources are read-oriented operations and may still involve server-side generation; prompts are reusable templates. If an MCP server is going to fall over under load, it will fall over on a tools/call — so that is the path measured here.

3.Methodology — A First Locust Script

Before the harness is generalised, the entire MCP life cycle is written by hand against one server. The target is the public Microsoft Learn MCP at https://learn.microsoft.com/api/mcp — no auth, stateful (it issues a session id on initialize), spec-compliant Streamable HTTP. Five blocks, each one of the four wire phases plus the data they operate on. Everything below could live in a single file with one import: Locust.

3.1 Step 1 — Define the data and target

A list of queries to rotate through, and the standard Locust FastHttpUser pointed at the host. mcp_path and protocol_version are constants reused in every request below. 

import json, random, time, itertools
from locust import FastHttpUser, constant, task

QUERIES = [
    "azure functions cold start",
    "entra id conditional access",
    "aks cluster autoscaler",
    "cosmos db throughput rules",
]

_id_counter = itertools.count(1)


class LearnUser(FastHttpUser):
    host             = "https://learn.microsoft.com"
    mcp_path         = "/api/mcp"
    protocol_version = "2025-06-18"
    session_id: str | None = None

    # pace each VU at one task per second
    wait_time = constant(1)

3.2 Step 2 — Initialize the session

Locust calls on_start once per virtual user. The harness POSTs a JSON-RPC initialize with the protocol version and a client name, then reads the Mcp-Session-Id header off the response — every subsequent request has to send it back. The server replies as Server-Sent Events, so the body is data: {...} framed; the JSON is plucked off the data: line. 

    def on_start(self):
        rpc_id = next(_id_counter)
        payload = {
            "jsonrpc": "2.0", "id": rpc_id, "method": "initialize",
            "params": {
                "protocolVersion": self.protocol_version,
                "capabilities": {},
                "clientInfo": {"name": "learn-first", "version": "1.0.0"},
            },
        }
        headers = {
            "Content-Type": "application/json",
            "Accept":       "application/json, text/event-stream",
        }
        with self.client.post(self.mcp_path, data=json.dumps(payload),
                              headers=headers, name="learn:initialize",
                              catch_response=True) as r:
            r.raise_for_status()
            self.session_id = r.headers.get("Mcp-Session-Id")
            if not self.session_id:
                r.failure("no Mcp-Session-Id"); return
            r.success()

        self._notify_initialized()
        self._tools_list()

3.3 Step 3 — Confirm the handshake and list tools

Two more POSTs to finish opening the session. The notification tells the server the client is ready (it returns 202 with no body). tools/list is what would be parsed to discover tool names dynamically — for this script the names are already known, so it is called purely to honour the life cycle and record the latency. 

    def _headers(self):
        return {
            "Content-Type":         "application/json",
            "Accept":               "application/json, text/event-stream",
            "MCP-Protocol-Version": self.protocol_version,
            "Mcp-Session-Id":       self.session_id,
        }

    def _notify_initialized(self):
        payload = {"jsonrpc": "2.0", "method": "notifications/initialized"}
        with self.client.post(self.mcp_path, data=json.dumps(payload),
                              headers=self._headers(),
                              name="learn:notifications/initialized",
                              catch_response=True) as r:
            r.success() if r.status_code == 202 else r.failure(f"got {r.status_code}")

    def _tools_list(self):
        rpc_id = next(_id_counter)
        payload = {"jsonrpc": "2.0", "id": rpc_id, "method": "tools/list"}
        with self.client.post(self.mcp_path, data=json.dumps(payload),
                              headers=self._headers(),
                              name="learn:tools/list",
                              catch_response=True) as r:
            r.raise_for_status(); r.success()

3.4 Step 4 — Call a tool under load

This is what repeats. Each @task is one tool call; the integer is the weight, so Locust picks each method 50% of the time. The wire shape is identical to tools/list, just with method tools/call and a params block carrying the tool name and arguments. The response comes back SSE-framed; the harness walks the data: lines to find the frame whose id matches the request. 

    def _call(self, tool: str, arguments: dict):
        rpc_id = next(_id_counter)
        payload = {
            "jsonrpc": "2.0", "id": rpc_id, "method": "tools/call",
            "params": {"name": tool, "arguments": arguments},
        }
        with self.client.post(self.mcp_path, data=json.dumps(payload),
                              headers=self._headers(),
                              name=f"learn:tools/call:{tool}",
                              catch_response=True) as r:
            r.raise_for_status()
            obj = self._parse_sse(r.content, rpc_id)
            if "error" in obj:
                r.failure(f"jsonrpc error: {obj['error']}"); return
            r.success()

    @staticmethod
    def _parse_sse(body: bytes, want_id: int) -> dict:
        for ev in body.decode("utf-8", "replace").split("\n\n"):
            data = "\n".join(ln[5:].lstrip() for ln in ev.splitlines()
                             if ln.startswith("data:"))
            if not data: continue
            obj = json.loads(data)
            if obj.get("id") == want_id: return obj
        return {}

    @task(1)
    def docs_search(self):
        self._call("microsoft_docs_search", {"query": random.choice(QUERIES)})

    @task(1)
    def code_sample_search(self):
        self._call("microsoft_code_sample_search", {"query": random.choice(QUERIES)})

3.5 Step 5 — Close the session cleanly

Locust calls on_stop when a VU is shutting down. The spec asks the client to terminate the session explicitly — one DELETE to the same path with the session id, that is it. 

    def on_stop(self):
        if not self.session_id: return
        with self.client.delete(self.mcp_path, headers=self._headers(),
                                name="learn:DELETE",
                                catch_response=True) as r:
            r.success() if r.status_code in (200, 202, 204, 404, 405) \
                else r.failure(f"got {r.status_code}")
        self.session_id = None

This compact sample treats 404 and 405 as non-fatal teardown responses so repeated local runs do not fail on cleanup quirks. For a strict stateful server benchmark, 405 should be investigated or counted separately because it means the endpoint did not accept the DELETE cleanup.

That is a complete, runnable script for one MCP server — initialize, notify, list, call, delete. Saved as learn_first.py. The only dependency is Locust itself: 

# with uv
uv pip install "locust>=2.31"

# or with pip in an active venv
python -m pip install "locust>=2.31"

Then run it — this will produce a six-row Locust table: 

# with uv
uv run locust -f learn_first.py --headless -u 5 -r 1 -t 30s

# or, inside an active venv
python -m locust -f learn_first.py --headless -u 5 -r 1 -t 30s

After 30 seconds, Locust prints a summary table. One row per request name, with counts and latency in milliseconds. The illustrative output below is from a small bring-up run; the production-scale numbers reported in §8 are from a different fifteen-minute run on Azure Load Testing and will not match line for line.

Five virtual users for thirty seconds drove 85 requests at ~2.9 RPS with zero failures. Each VU sleeps one second between tasks (wait_time = constant(1)), so the rate is paced rather than going as fast as the network allows. The five lifecycle endpoints (initialize, notifications/initialized, tools/list, DELETE) each ran exactly once per VU; the rest is tools/call, split roughly 50/50 between the two @task methods.

4.Authentication Patterns

Before generalising the script, one more wrinkle: different MCP servers authenticate differently. Three patterns cover all four targets used in this guide, and each one stays a one- or two-line change on the user class.

4.1 Pattern A — no auth or optional bearer/API key (Learn, Context7)

Microsoft Learn can be called without a bearer token. Context7 is stateless in this harness and runs without extra auth, or with an optional bearer/API key for quota and reliability. When no header is needed, the class does not declare an auth_headers attribute and the base sends the standard MCP headers untouched.

4.2 Pattern B — static bearer from env (GitHub)

The token lives in GITHUB_MCP_TOKEN (a fine-grained PAT locally, a Key Vault reference in Azure Load Testing). It is read once at import time and stamped on every request via the class attribute: 

class GitHubUser(MCPUserBase):
    auth_headers = {
        "Authorization": f"Bearer {os.environ.get('GITHUB_MCP_TOKEN', '')}",
    }

4.3 Pattern C — dynamic token per request (ADO Remote)

The Azure DevOps Remote MCP is stateless and uses Entra bearer auth. Azure DevOps tokens expire roughly every hour, so a static value is not suitable for a long run. The class overrides _extra_headers() — a hook the base calls before every POST — and mints a fresh token on demand: 

class ADOUser(StatelessMCPUser):
    auth_headers = {"X-MCP-Readonly": "true"}

    def _extra_headers(self) -> dict[str, str]:
        return {
            **self.auth_headers,
            "Authorization": f"Bearer {_get_ado_token()}",
        }

Locally _get_ado_token() is backed by DefaultAzureCredential (cached az login); the SDK auto-refreshes. In Azure Load Testing the same function short-circuits to a pre-minted token injected from Key Vault. The user class does not care which mode is active.

That is the whole auth surface: one class attribute for static headers, one method override for dynamic ones. Everything else — sessions, retries, timing — lives in the base, which is the next section.

5.Framework Architecture

Source code

Everything in this section — _base.py, learn.py, github.py, context7.py, ado.py, multi.py, locust.conf, and the Azure Load Testing YAML — lives in the public repo I maintain: github.com/kroy92/azure-load-test-mcp-server.

learn_first.py works, but every MCP server in this study follows the same five-block flow — initialize, notify, list, call, and DELETE only for session-bearing users. Only the host, the tool names, session posture, and auth headers change. The wire plumbing is therefore lifted into a shared base class, leaving each per-server file with only the configuration and tasks that genuinely differ.

Why a flat-file framework?

The harness keeps _base.py, one server file per MCP endpoint, multi.py, and the config files in a flat project root so Azure Load Testing can upload the same file set directly. That shape avoids packaging steps, keeps each server easy to inspect, and makes local and cloud runs use the same imports.

The same framework is useful before rolling out a custom MCP server: add one user class for the new endpoint, import it from multi.py, and benchmark the server's tool mix locally and in Azure Load Testing before enabling broader agent traffic.

MCPUserBase in _base.py owns the life cycle: on_start runs _initialize, _notify_initialized, and _tools_list; on_stop runs _terminate_session; _call wraps _jsonrpc for every tools/call; wait_time implements the agent-shaped 80/20 think time; _extra_headers is the hook subclasses override to mint auth tokens. Session-bearing users in this harness (Learn as observed in the demo run, and GitHub) inherit MCPUserBase directly. Stateless servers (Context7, ADO Remote) go through StatelessMCPUser, which sets requires_session = False so the DELETE step is a no-op. Each leaf class is then a small set of @task methods.

With that base in place, each per-server file is a list of queries, five config constants, and a handful of @task methods. multi.py is twenty lines of imports; Locust loads each User subclass and spawns its declared fixed_count, so one process drives all four servers in parallel. The same files run unchanged in Azure Load Testing via azure-loadtest.yaml — the engines just need the same env vars.

6.Deployment — Local and Cloud Execution

Four user classes, but the goal is one Locust process driving all of them — and the same files running in Azure Load Testing without edits. multi.py is the entrypoint. It is twenty lines of imports; Locust discovers each User subclass and spawns its declared fixed_count: 

from ado import ADOUser            # noqa: F401
from context7 import Context7User  # noqa: F401
from github import GitHubUser      # noqa: F401
from learn import LearnUser        # noqa: F401

The four fixed_count values must sum to users in locust.conf (default 2 + 5 + 5 + 5 = 17), otherwise Locust warns and rebalances. Run it locally: 

$ uv run locust -f multi.py --config locust.conf

Same files, no changes — ship them to Azure Load Testing through a one-page YAML. Azure Load Testing uploads the project, spins up engine VMs (1–45), runs the same locust -f multi.py behind the scenes, and streams results back to a portal dashboard:

testId: mcp-multi-flat
displayName: MCP multi-server cloud smoke
testPlan: multi.py
testType: Locust

  - _base.py
  - mcp_client.py
  - github.py
  - learn.py
  - context7.py
  - ado.py
  - requirements.txt
  # … plus query data files

properties:
  userPropertyFile: locust.conf

env:
  - { name: LOCUST_USERS,    value: "17" }
  - { name: GITHUB_USERS,    value: "2"  }
  - { name: LEARN_USERS,     value: "5"  }
  - { name: CONTEXT7_USERS,  value: "5"  }
  - { name: ADO_USERS,       value: "5"  }
  - { name: ADO_ORG,         value: "<your-ado-org>" }
  - { name: ADO_PROJECT,     value: "<your-ado-project>" }

# Bearer tokens stay in Key Vault; Azure Load Testing’s system-assigned identity
# resolves them at run time and injects as env vars.
secrets:
  - { name: GITHUB_MCP_TOKEN,  value: https://<your-keyvault>.vault.azure.net/secrets/github-mcp-token }
  - { name: ADO_MCP_TOKEN,     value: https://<your-keyvault>.vault.azure.net/secrets/ado-mcp-token }
  - { name: CONTEXT7_API_KEY,  value: https://<your-keyvault>.vault.azure.net/secrets/context7-api-key }

failureCriteria:
  - avg(response_time_ms) > 8000
  - percentage(error) > 25

The test is submitted with the Azure CLI. Azure Load Testing pulls the secrets from Key Vault via its system-assigned identity at run time, so the only thing required locally is an az login against the subscription that owns the load-testing resource: 

$ az login
$ az load test update \
    --load-test-resource <your-loadtest-resource> \
    --resource-group <your-rg> \
    --test-id mcp-multi-flat \
    --load-test-config-file azure-loadtest.yaml
$ az load test-run create \
    --load-test-resource <your-loadtest-resource> \
    --resource-group <your-rg> \
    --test-id mcp-multi-flat \
    --test-run-id run-$(date +%Y%m%d-%H%M%S) \
    --no-wait

One wrinkle worth calling out: the ADO Remote MCP server uses Entra ID and its tokens expire in about an hour. If the Azure DevOps organisation lives in a different tenant from the subscription that hosts Azure Load Testing, before each run the operator needs to az login against the ADO tenant, request an access token for the Azure DevOps resource, and write it to the ado-mcp-token Key Vault secret. Same tenant for both? Skip it entirely — when Locust runs locally, az login reads the secrets from Key Vault, and when Azure Load Testing runs the test in the cloud the load-test resource's managed identity does the same thing.

Three design choices keep the run reproducible. Auth: secret values are resolved from Key Vault at run time and are not stored in source files or the YAML definition. Azure Load Testing reads them via managed identity, and the user classes read the same env-var names locally and in cloud. Failure criteria turn the YAML into a pass/fail gate suitable for a release pipeline. Engine count is the scaling setting: increase engineInstances to spread the same VU mix across more workers.

7.Experimental Setup

Same four files, real endpoints. Before the run output, this section describes what one Locust process actually puts on the wire when launched.

7.1 Workload model

Seventeen virtual users split four ways, each VU picking tools from its server's @task weights, paced by the agent-shaped think time (80% short intra-turn pauses 0.1–0.5 s, 20% inter-turn idles 5–30 s).

7.2 Local validation

Before sending the workload to the cloud, run it in one Python process locally. locust.conf pins the settings (and is the same file Azure Load Testing consumes via userPropertyFile): 

# locust.conf
locustfile  = multi.py
headless    = true
users       = 17        # 2 GitHub + 5 Learn + 5 Context7 + 5 ADO
spawn-rate  = 5
run-time    = 15m
csv         = results/run
html        = results/run.html
$ locust --config locust.conf --run-time 2m \
    --csv results/local-2m --html results/local-2m.html

Useful for catching a broken assertion or an expired token in two minutes instead of fifteen. The headline run reported in this guide is the cloud-path execution submitted to Azure Load Testing.

9.Discussion

9.1 Latency signatures and what they may suggest

The four servers fall into three rough tiers. Microsoft Learn is the fastest (sub-900 ms p90 on both tools), which may reflect cached, anonymous content delivery. The Azure DevOps Remote MCP is the most consistent (a tight gap between average and p90 across all five tools), which may reflect a stable internal service backed by deterministic queries. Context7 is the slowest (over 1 s on every call), which may reflect a synchronous embedding lookup per request. GitHub sits in the middle, but search_code is bimodal — a 2.8× average-to-p90 ratio and the run's only failures — which may reflect API-level rate limiting.

9.2 The handshake is not free

Cold-session cost varies tenfold across the four servers, from 44 ms (Learn tools/list) to 335 ms (ADO initialize). For agents that open and close many short-lived sessions, this overhead matters. The result argues for session reuse where possible — though that is a property of the agent runtime, not of MCP itself.

9.3 Limitations

  • Single-engine run. The reported run uses one Azure Load Testing engine instance. Higher concurrency (engineInstances > 1) was not exercised in this guide and would be required to characterise saturation behaviour.
  • Modest VU count. Seventeen virtual users is a smoke-grade workload, not a stress test. The numbers reported are a baseline at light load, not a capacity ceiling.
  • Non-replayable third-party endpoints. Three of the four servers are external services with their own rate limits, terms of service, and operational variability. Numbers reported here describe the state of those services on the run date and should not be treated as service-level commitments.
  • Tools only. The harness exercises tools/call exclusively. Resources and prompts use the same wire format but were not measured.

9.4 Operational considerations

Load testing third-party hosted MCP servers should be done with permission and within the operator's published rate limits. The targets used in this guide are public endpoints; the workload is deliberately small; and any team replicating this work against services they do not own should review terms of service first.

10.Conclusion

A protocol-faithful load-testing harness for hosted MCP servers can be built in a few hundred lines of Python on top of Locust. The key design choices — a stateful/stateless split at the base class, an auth hook that supports static and dynamic tokens, and a response reader that handles both application/json and text/event-stream — make the harness portable across the hosted MCP servers exercised in this study and plausibly across most others in the same shape.

The same project files run unchanged from a developer laptop and from Azure Load Testing engines in the cloud, with secrets resolved through Key Vault by managed identity at run time. The run reported here surfaces distinct, interpretable latency signatures across four production servers and demonstrates that the harness is suitable both for development-time smoke checks and for pass/fail gates in release pipelines via failureCriteria.

Future work could extend the harness to cover the resources and prompts primitives, introduce ramp-and-soak workload profiles, and use multi-engine fan-out to characterise saturation behaviour on production-scale targets.
