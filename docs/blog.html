Load testing hosted MCP servers with Locust

MCP servers are everywhere. Every team is exposing the tools, data, and actions their product can perform so an AI agent can reach in and use them. An agent is just an LLM running in a loop — it looks, thinks, calls a tool, looks again, until the job is done. The tools live on MCP.

MCP gives every agent and every tool the same way to talk. Instead of writing a fresh, fragile integration for each new service, a team builds one MCP server and any agent can use it. Each server offers three simple kinds of things: tools the agent can call to do something, resources it can read for context, and prompts it can reuse. That small, shared vocabulary is what lets one agent reach across many servers — docs, code, your product — without custom glue for each one.

1.Approach

Hosted means HTTP, not stdio

MCP defines two transports: stdio, where the agent spawns the server as a local subprocess and pipes JSON-RPC over its stdin and stdout, and Streamable HTTP, where the server lives on the network. This post focuses on Streamable HTTP because that's the only transport with anything to load test — a stdio server has no socket, no concurrency of its own, and is bounded by the single agent process that launched it.

The MCP life cycle

Every agent-to-server conversation follows the same formal flow: how the connection opens, how messages are exchanged while it's alive, and how it shuts down cleanly. That flow is the MCP life cycle, and our load test has to mimic it end to end — otherwise we're not testing what a real agent actually puts on the wire.

Phase 1 — The handshake

Throughout the rest of this post the server is Microsoft Learn, hosted at https://learn.microsoft.com/api/mcp. Every arrow you see in the diagrams is a POST to that URL.

Before the agent can call a single tool, it has to introduce itself and let the server introduce itself back. Three messages, in order, every time:

The session id is the only thing the agent has to hold onto. Every subsequent tools/call rides on the same Mcp-Session-Id header until the connection ends — which is exactly what a Locust virtual user has to do too.

One caveat: not every MCP server mints a session. Stateless servers (GitHub's hosted MCP, Context7, plenty of read-only wrappers) skip the Mcp-Session-Id header entirely — the handshake still happens, but every tools/call after it is a standalone POST. Both shapes are valid; the client just has to handle either.

Phase 2 — Discover what the server can do

Once the handshake is done, the agent doesn't yet know what tools the server actually exposes. It asks. The answer is the menu the LLM gets to pick from on every turn.

The response is a flat list of tool descriptors — name, human-readable description, and a JSON Schema for the arguments. A real agent feeds this list straight into the LLM's system prompt so the model knows what it can call. A load test doesn't need the LLM in the loop, but it still needs to call tools/list at least once per session: it's how the harness learns the real tool names instead of hard-coding them.

Phase 3 — Call a tool

This is the request the entire load test exists to measure. The agent picks a tool from the menu, fills in the arguments, and POSTs a tools/call. The reply comes back in one of two shapes — the client asks for both with Accept: application/json, text/event-stream and the server picks one per request.

The request is ordinary JSON-RPC:

Mode A — application/json. The body is the JSON-RPC payload. One response, no wrapping.

Mode B — text/event-stream. The body is a stream of frames. Each frame is two lines — event: message and data: <json> — and the server can send several before closing.

Same JSON-RPC result either way. The difference is the wrapper: application/json hands you one body to parse; text/event-stream hands you a stream you have to read, split into frames, and pull the data: values from. Calling .json() on an SSE response will throw. Pick one mode and the harness works against some servers and breaks against others. So always check Content-Type first, then choose how to read the body.

Phase 4 — Clean up

When the agent is done, it tells the server. One DELETE with the session id, and the server tears the session down.

Stateless servers don't need this — there's no session to delete, so the agent just stops sending requests. For stateful servers, skipping the DELETE isn't fatal (sessions eventually time out) but it leaks server memory while they wait. A polite client always cleans up.

The other two primitives — in one breath

MCP defines two more primitives alongside tools: resources (read-only data the agent can pull into context) and prompts (reusable templates the user can invoke). Both follow the same JSON-RPC + JSON/SSE shape you've already seen.

This post focuses on tools only. Tools are where the real work happens — they're the only primitive that runs server-side code, hits databases, calls downstream APIs, and actually does something. Resources are static reads; prompts are templates. If your MCP server is going to fall over under load, it will fall over on a tools/call.

2.Writing the first Locust script

Before we factor anything out, let's write the whole MCP life cycle by hand against one server. The target is the public Microsoft Learn MCP at https://learn.microsoft.com/api/mcp — no auth, stateful (it issues a session id on initialize), spec-compliant Streamable HTTP. Five blocks, each one of the four wire phases plus the data we feed it. Everything below could live in a single file with one import: Locust.

Step 1. Define the data and target

A list of queries to rotate through, and the standard Locust FastHttpUser pointed at the host. mcp_path and protocol_version are constants we'll reuse in every request below. 

import json, random, time, itertools
from locust import FastHttpUser, constant, task

QUERIES = [
    "azure functions cold start",
    "entra id conditional access",
    "aks cluster autoscaler",
    "cosmos db throughput rules",
]

_id_counter = itertools.count(1)


class LearnUser(FastHttpUser):
    host             = "https://learn.microsoft.com"
    mcp_path         = "/api/mcp"
    protocol_version = "2025-06-18"
    session_id: str | None = None

    # pace each VU at one task per second
    wait_time = constant(1)

Step 2. Initialize the session

Locust calls on_start once per virtual user. We POST a JSON-RPC initialize with our protocol version and a client name, then read the Mcp-Session-Id header off the response — every subsequent request has to send it back. The server replies as Server-Sent Events, so the body is data: {...} framed; we pluck the JSON off the data: line. 

    def on_start(self):
        rpc_id = next(_id_counter)
        payload = {
            "jsonrpc": "2.0", "id": rpc_id, "method": "initialize",
            "params": {
                "protocolVersion": self.protocol_version,
                "capabilities": {},
                "clientInfo": {"name": "learn-first", "version": "1.0.0"},
            },
        }
        headers = {
            "Content-Type": "application/json",
            "Accept":       "application/json, text/event-stream",
        }
        with self.client.post(self.mcp_path, data=json.dumps(payload),
                              headers=headers, name="learn:initialize",
                              catch_response=True) as r:
            r.raise_for_status()
            self.session_id = r.headers.get("Mcp-Session-Id")
            if not self.session_id:
                r.failure("no Mcp-Session-Id"); return
            r.success()

        self._notify_initialized()
        self._tools_list()

Step 3. Confirm the handshake and list available tools

Two more POSTs to finish opening the session. The notification tells the server we're ready (it returns 202 with no body). tools/list is what we'd parse to discover tool names dynamically — for this script we already know them, so we just call it to honour the life cycle and record the latency. 

    def _headers(self):
        return {
            "Content-Type":         "application/json",
            "Accept":               "application/json, text/event-stream",
            "MCP-Protocol-Version": self.protocol_version,
            "Mcp-Session-Id":       self.session_id,
        }

    def _notify_initialized(self):
        payload = {"jsonrpc": "2.0", "method": "notifications/initialized"}
        with self.client.post(self.mcp_path, data=json.dumps(payload),
                              headers=self._headers(),
                              name="learn:notifications/initialized",
                              catch_response=True) as r:
            r.success() if r.status_code == 202 else r.failure(f"got {r.status_code}")

    def _tools_list(self):
        rpc_id = next(_id_counter)
        payload = {"jsonrpc": "2.0", "id": rpc_id, "method": "tools/list"}
        with self.client.post(self.mcp_path, data=json.dumps(payload),
                              headers=self._headers(),
                              name="learn:tools/list",
                              catch_response=True) as r:
            r.raise_for_status(); r.success()

Step 4. Call a tool under load

This is what repeats. Each @task is one tool call; the integer is the weight, so Locust picks each method 50% of the time. The wire shape is identical to tools/list, just with method tools/call and a params block carrying the tool name and arguments. The response comes back SSE-framed; we walk the data: lines to find the frame whose id matches our request. 

    def _call(self, tool: str, arguments: dict):
        rpc_id = next(_id_counter)
        payload = {
            "jsonrpc": "2.0", "id": rpc_id, "method": "tools/call",
            "params": {"name": tool, "arguments": arguments},
        }
        with self.client.post(self.mcp_path, data=json.dumps(payload),
                              headers=self._headers(),
                              name=f"learn:tools/call:{tool}",
                              catch_response=True) as r:
            r.raise_for_status()
            obj = self._parse_sse(r.content, rpc_id)
            if "error" in obj:
                r.failure(f"jsonrpc error: {obj['error']}"); return
            r.success()

    @staticmethod
    def _parse_sse(body: bytes, want_id: int) -> dict:
        for ev in body.decode("utf-8", "replace").split("\n\n"):
            data = "\n".join(ln[5:].lstrip() for ln in ev.splitlines()
                             if ln.startswith("data:"))
            if not data: continue
            obj = json.loads(data)
            if obj.get("id") == want_id: return obj
        return {}

    @task(1)
    def docs_search(self):
        self._call("microsoft_docs_search", {"query": random.choice(QUERIES)})

    @task(1)
    def code_sample_search(self):
        self._call("microsoft_code_sample_search", {"query": random.choice(QUERIES)})

Step 5. Close the session cleanly

Locust calls on_stop when a VU is shutting down. The spec asks the client to terminate the session explicitly — one DELETE to the same path with the session id, that's it. 

    def on_stop(self):
        if not self.session_id: return
        with self.client.delete(self.mcp_path, headers=self._headers(),
                                name="learn:DELETE",
                                catch_response=True) as r:
            r.success() if r.status_code in (200, 202, 204, 404, 405) \
                else r.failure(f"got {r.status_code}")
        self.session_id = None

That's a complete, runnable script for one MCP server — initialize, notify, list, call, delete. Save it as learn_first.py. The only dependency is Locust itself: 

# with uv
uv pip install "locust>=2.31"

# or with pip in an active venv
python -m pip install "locust>=2.31"

Then run it — this will produce the same six-row Locust table you'll see in §4: 

# with uv
uv run locust -f learn_first.py --headless -u 5 -r 1 -t 30s

# or, inside an active venv
python -m locust -f learn_first.py --headless -u 5 -r 1 -t 30s

After 30 seconds, Locust prints the summary table. One row per request name, with counts and latency in milliseconds:

Endpoint # reqs fails p50 p95
learn:initialize 5 0 820 850
learn:notifications/initialized 5 0 620 660
learn:tools/list 5 0 320 330
learn:tools/call:microsoft_docs_search 36 0 880 1500
learn:tools/call:microsoft_code_sample_search 29 0 860 1500
learn:DELETE 5 0 380 510

Five virtual users for thirty seconds drove 85 requests at ~2.9 RPS with zero failures. Each VU sleeps one second between tasks (wait_time = constant(1)), so the rate is paced rather than going as fast as the network allows. The five lifecycle endpoints (initialize, notifications/initialized, tools/list, DELETE) each ran exactly once per VU; the rest is tools/call, split roughly 50/50 between the two @task methods. We'll come back to these numbers in §5 when we talk about what they mean.

3.Handling auth

Before we generalise the script we need one more wrinkle: different MCP servers authenticate differently. Three patterns cover all four targets, and each one stays a one- or two-line change on the user class.

Pattern A — no auth (Learn, Context7)

The endpoint is public. Nothing to set. The class doesn’t declare an auth_headers attribute at all and the base sends the standard MCP headers untouched.

Pattern B — static bearer from env (GitHub)

The token lives in GITHUB_MCP_TOKEN (a fine-grained PAT locally, a Key Vault reference in Azure Load Testing). It’s read once at import time and stamped on every request via the class attribute: 

class GitHubUser(MCPUserBase):
    auth_headers = {
        "Authorization": f"Bearer {os.environ.get('GITHUB_MCP_TOKEN', '')}",
    }

Pattern C — dynamic token per request (ADO Remote)

Azure DevOps tokens expire roughly every hour, so a static value won’t survive a long run. The class overrides _extra_headers() — a hook the base calls before every POST — and mints a fresh token on demand: 

class ADOUser(StatelessMCPUser):
    auth_headers = {"X-MCP-Readonly": "true"}

    def _extra_headers(self) -> dict[str, str]:
        return {
            **self.auth_headers,
            "Authorization": f"Bearer {_get_ado_token()}",
        }

Locally _get_ado_token() is backed by DefaultAzureCredential (cached az login); the SDK auto-refreshes. In Azure Load Testing the same function short-circuits to a pre-minted token injected from Key Vault. The user class doesn’t care which mode is active.

That’s the whole auth surface: one class attribute for static headers, one method override for dynamic ones. Everything else — sessions, retries, timing — lives in the base, which is the next section.

4.Framework

Grab the code

Everything in this section — _base.py, learn.py, github.py, context7.py, ado.py, multi.py, locust.conf, and the Azure Load Testing YAML — lives in the public repo: github.com/kroy92/azure-load-test-mcp-server. Clone it and follow along.

learn_first.py works, but every other MCP server we want to test does the same five-block dance — initialize, notify, list, call, delete. Only the host, the tool names, and (for some) the auth header change. So we lifted the wire plumbing into a shared base class and let each per-server file keep just the bits that actually differ.

MCPUserBase in _base.py owns the life cycle: on_start runs _initialize, _notify_initialized, and _tools_list; on_stop runs _terminate_session; _call wraps _jsonrpc for every tools/call; wait_time implements the agent-shaped 80/20 think time; _extra_headers is the hook subclasses override to mint auth tokens. Stateful servers (Learn, GitHub) inherit MCPUserBase directly. Stateless servers (Context7, ADO Remote) go through StatelessMCPUser, which just flips requires_session = False so the DELETE step is a no-op. Each leaf class is then a handful of @task methods.

With that base in place, each per-server file is a list of queries, five config constants, and a handful of @task methods. multi.py is twenty lines of imports; Locust loads each User subclass and spawns its declared fixed_count, so one process drives all four servers in parallel. The same files run unchanged in Azure Load Testing via azure-loadtest.yaml — the engines just need the same env vars.

5.One process, four servers — locally and in the cloud

Four user classes, but we want one Locust process driving all of them — and we want the same files to run in Azure Load Testing without edits. multi.py is the entrypoint. It’s twenty lines of imports; Locust discovers each User subclass and spawns its declared fixed_count: 

from ado import ADOUser            # noqa: F401
from context7 import Context7User  # noqa: F401
from github import GitHubUser      # noqa: F401
from learn import LearnUser        # noqa: F401

The four fixed_count values must sum to users in locust.conf (default 2 + 5 + 5 + 5 = 17), otherwise Locust warns and rebalances. Run it locally: 

$ uv run locust -f multi.py --config locust.conf

Same files, no changes — ship them to Azure Load Testing through a one-page YAML. Azure Load Testing uploads the project, spins up engine VMs (1–45), runs the same locust -f multi.py behind the scenes, and streams results back to a portal dashboard:

  - percentage(error) > 25

Submit it with the Azure CLI. Azure Load Testing pulls the secrets from Key Vault via its system-assigned identity at run time, so the only thing you need locally is an az login against the subscription that owns the load-testing resource: 

$ az login
$ az load test update \
    --load-test-resource <your-loadtest-resource> \
    --resource-group <your-rg> \
    --test-id mcp-multi-flat \
    --load-test-config-file azure-loadtest.yaml
$ az load test-run create \
    --load-test-resource <your-loadtest-resource> \
    --resource-group <your-rg> \
    --test-id mcp-multi-flat \
    --test-run-id run-$(Get-Date -Format yyyyMMdd-HHmmss) \
    --no-wait

One wrinkle worth calling out: the ADO Remote MCP server uses Entra ID and its tokens expire in about an hour. If your Azure DevOps organisation lives in a different tenant from the subscription that hosts Azure Load Testing, before each run you need to az login against the ADO tenant, request an access token for the Azure DevOps resource, and write it to the ado-mcp-token Key Vault secret. Same tenant for both? Skip it entirely — when you run Locust on your laptop your az login reads the secrets from Key Vault, and when Azure Load Testing runs the test in the cloud the load-test resource's managed identity does the same thing.

Three things keep this honest. Auth: secrets never leave Key Vault — Azure Load Testing pulls them via managed identity and the user classes read the same env-var names locally and in cloud. Failure criteria turn the YAML into a pass/fail gate suitable for a release pipeline. Engine count is the only knob that changes when you scale: bump engineInstances to spread the same VU mix across more workers.

6.Running a demo test

Same four files, real endpoints. Before we look at the run output, here’s what one Locust process actually puts on the wire when you launch it.

Workload model

Seventeen virtual users split four ways, each VU picking tools from its server’s @task weights, paced by the agent-shaped think time (80% short intra-turn pauses 0.1–0.5 s, 20% inter-turn idles 5–30 s).

Trying it locally first

Before sending it to the cloud, the same workload runs in one Python process on your laptop. locust.conf pins the knobs (and is the same file Azure Load Testing consumes via userPropertyFile): 

# locust.conf
locustfile  = multi.py
headless    = true
users       = 17        # 2 GitHub + 5 Learn + 5 Context7 + 5 ADO
spawn-rate  = 5
run-time    = 15m
csv         = results/run
html        = results/run.html
$ locust --config locust.conf --run-time 2m \
    --csv results/local-2m --html results/local-2m.html

Useful for catching a broken assertion or an expired token in two minutes instead of fifteen. For this post though, the run that matters is the one that proves the cloud path works — so let's submit it to Azure Load Testing.
