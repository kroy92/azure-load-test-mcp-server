4. VU lifecycle (sequence diagram)

Each Locust virtual user (VU) now follows exactly this flow against its assigned server:

sequenceDiagram autonumber participant VU as Locust VU participant S as MCP Server Note over VU,S: on_start — once per VU VU->>S: POST /mcp initialize {protocolVersion, capabilities, clientInfo} S-->>VU: 200 + Mcp-Session-Id header + result VU->>S: POST /mcp notifications/initialized (no id) S-->>VU: 202 Accepted VU->>S: POST /mcp tools/list S-->>VU: 200 result.tools[] Note over VU,S: @task loop until -t expires loop VU->>S: POST /mcp tools/call {name, arguments} S-->>VU: 200 result (text / structuredContent) end

Every request carries the same Mcp-Session-Id header obtained at step 2, plus MCP-Protocol-Version on every call after initialize. Each step appears in stats labelled <server_label>:<method> (e.g. github:tools/call:search_repositories).

5. Streamable HTTP transport

Streamable HTTP is the only MCP transport this harness supports. It is a single-endpoint design where every JSON-RPC message goes over POST <mcp_path>.

flowchart TB C[Client / Locust VU] S[MCP Server] C ==>|POST mcp_path
Accept: application/json, text/event-stream
Mcp-Session-Id, MCP-Protocol-Version| S S -->|Content-Type: application/json
single JSON-RPC envelope| C S -->|Content-Type: text/event-stream
SSE events ending with the response| C classDef c fill:#e0f2fe,stroke:#0284c7,color:#075985; classDef s fill:#ede9fe,stroke:#7c3aed,color:#5b21b6; class C c class S s

The server picks one of the two response framings per request. The client must accept both — which is why the harness sends Accept: application/json, text/event-stream on every call and dispatches in _jsonrpc():

ctype = (resp.headers.get("Content-Type") or "").lower()
body  = resp.content
if "text/event-stream" in ctype:
    obj = parse_sse_for_id(body, rpc_id)   # SSE branch
elif "application/json" in ctype:
    obj = json.loads(body) if body else {} # JSON branch
else:
    resp.failure(f"unexpected content-type: {ctype!r}")

6. application/json vs text/event-stream

application/json

One body, one JSON object, done.

HTTP/1.1 200 OK
Content-Type: application/json

{"jsonrpc":"2.0","id":1,
 "result":{ "protocolVersion":"2025-06-18", ... }}
  • Simplest framing — works like a normal REST call.
  • Server cannot interleave notifications, progress, or sampling requests.
  • Lower overhead for tiny responses.

text/event-stream (SSE)

A sequence of events; the JSON-RPC reply arrives in one data: line.

HTTP/1.1 200 OK
Content-Type: text/event-stream
Transfer-Encoding: chunked

event: message
data: {"jsonrpc":"2.0","id":1,
       "result":{ "protocolVersion":"2025-06-18", ... }}
  • Server can stream progress / log / server→client requests before the final response.
  • Required for unsolicited messages (sampling, elicitation).
  • Slightly more overhead; some HTTP/1.1 proxies buffer it.

Side-by-side wire diagram

sequenceDiagram participant C as Client participant J as Server (JSON mode) participant E as Server (SSE mode) C->>J: POST /mcp tools/call J-->>C: 200 application/json
{"jsonrpc":"2.0","id":N,"result":...} C->>E: POST /mcp tools/call E-->>C: 200 text/event-stream E-->>C: event: message
data: {progress 25%} E-->>C: event: message
data: {progress 75%} E-->>C: event: message
data: {"jsonrpc":"2.0","id":N,"result":...}

In both cases the client receives exactly one final JSON-RPC response with the matching id. SSE just allows extra messages to flow first.
